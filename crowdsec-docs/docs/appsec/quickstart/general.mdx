# CrowdSec WAF General Setup

This guide covers the core AppSec Component setup that applies to all web servers and reverse proxies. After completing these steps, configure your remediation component (bouncer) to forward requests to the AppSec Component.

## Prerequisites

- **CrowdSec Security Engine** (>= 1.5.6) installed and running
- A compatible remediation component (bouncer) for your web server or reverse proxy

## AppSec Component Setup

AppSec setup has two steps:
- Download rules and configuration collections.
- Configure AppSec as a new acquisition datasource ([AppSec datasource](/log_processor/data_sources/appsec.md)).
  
The following sections will guide you through the default setup.

After installation, verify everything works with the [🩺 Health Check](/u/getting_started/health_check).

### Collection Installation

Install the essential AppSec collections that provide virtual patching rules and generic attack detection:

```bash
sudo cscli collections install crowdsecurity/appsec-virtual-patching crowdsecurity/appsec-generic-rules
```

These collections include:
- **Virtual Patching Rules**: Protection against known vulnerabilities (CVEs)
- **Generic Attack Detection**: Common web attack patterns
- **AppSec Configuration**: Default [AppSec configuration file](/appsec/configuration.md) linking rules together
- **CrowdSec Parsers & Scenarios**: For processing AppSec events and creating alerts

### Acquisition Configuration

Configure CrowdSec to expose the AppSec Component by creating an acquisition file ([AppSec datasource](/log_processor/data_sources/appsec.md)).

1. Create the acquisition directory (if it doesn't exist):

```bash
sudo mkdir -p /etc/crowdsec/acquis.d/
```

2. Create the AppSec acquisition configuration:

```bash
sudo cat > /etc/crowdsec/acquis.d/appsec.yaml << EOF
appsec_configs:
  - crowdsecurity/appsec-default
labels:
  type: appsec
listen_addr: 127.0.0.1:7422
source: appsec
name: myAppSecComponent
EOF
```

**Configuration explained:**
- `appsec_configs`: Uses the [AppSec configuration(s)](/appsec/configuration.md) from the installed collections
- `listen_addr`: IP and port where the AppSec Component listens (default: 127.0.0.1:7422)
- `source`: Identifies this as an AppSec data source
- `name`: A friendly name for your AppSec component

:::warning Security Note
Do not expose the AppSec Component to the internet. It should only be accessible from your web server or reverse proxy.
:::

### Choose Your Protection Level

Not sure which rules to use? Pick the profile that matches your needs:

<TabItem value="strict">

**Virtual patching only — zero false positives, instant protection**

Best for: New users, production environments, getting started quickly.

Blocks actively exploited CVEs and common attack vectors. Updated daily. Will never block legitimate traffic.

```bash
sudo cscli collections install crowdsecurity/appsec-virtual-patching crowdsecurity/appsec-generic-rules
```

```yaml title="/etc/crowdsec/acquis.d/appsec.yaml"
appsec_configs:
  - crowdsecurity/appsec-default
labels:
  type: appsec
listen_addr: 127.0.0.1:7422
source: appsec
name: myAppSecComponent
```

</TabItem>

<TabItem value="balanced">

**Virtual patching + OWASP CRS detection (out-of-band)**

Best for: Users who want broader attack detection while keeping zero blocking false positives.

Adds OWASP CRS rules in *out-of-band* mode: attacks are detected and logged, and repeat offenders are banned after 5+ violations — but individual requests are never blocked on the first hit.

```bash
sudo cscli collections install crowdsecurity/appsec-virtual-patching crowdsecurity/appsec-generic-rules crowdsecurity/appsec-crs
```

```yaml title="/etc/crowdsec/acquis.d/appsec.yaml"
appsec_configs:
  - crowdsecurity/appsec-default    # Virtual patching (in-band blocking)
  - crowdsecurity/crs               # OWASP CRS (out-of-band detection + ban)
labels:
  type: appsec
listen_addr: 127.0.0.1:7422
source: appsec
name: myAppSecComponent
```

</TabItem>

<TabItem value="paranoid">

**Full OWASP CRS in blocking mode**

Best for: Security-focused environments willing to tune for false positives.

Immediately blocks any request matching OWASP CRS rules. Provides maximum protection but requires testing and tuning in your environment before production use.

```bash
sudo cscli collections install crowdsecurity/appsec-virtual-patching crowdsecurity/appsec-generic-rules crowdsecurity/appsec-crs-inband
```

```yaml title="/etc/crowdsec/acquis.d/appsec.yaml"
appsec_configs:
  - crowdsecurity/appsec-default    # Virtual patching (in-band blocking)
  - crowdsecurity/crs-inband        # OWASP CRS (in-band blocking)
labels:
  type: appsec
listen_addr: 127.0.0.1:7422
source: appsec
name: myAppSecComponent
```

:::warning
Always test in-band CRS in a staging environment first. See [WAF Deployment Strategies](/appsec/advanced_deployments.mdx) for tuning guidance.
:::

</TabItem>
</Tabs>

:::info Start with Strict
If you're unsure, start with **Strict**. It's the safest choice: you get real protection immediately, and you can upgrade to a broader profile once you're comfortable with the setup.
:::

### Start the AppSec Component

Restart CrowdSec to activate the AppSec Component:

```bash
sudo systemctl restart crowdsec
```

### Attack Simulation — See the WAF in Action

Once your AppSec-capable bouncer is configured, simulate real attack traffic to confirm the WAF is working end-to-end.

#### Test 1: Connectivity check (no rules required)

Trigger the built-in `crowdsecurity/appsec-generic-test` probe. This does **not** block traffic — it just confirms the AppSec Component is reachable from your bouncer.

```bash
curl -I https://<your-service-url>/crowdsec-test-NtktlJHV4TfBSK3wvlhiOBnl
```

Confirm the probe was seen:

```bash
sudo cscli alerts list | grep crowdsecurity/appsec-generic-test
```

The alert also appears in the [CrowdSec Console](https://app.crowdsec.net):

![appsec-generic-test console view](/img/appsec-generic-test-console.png)

:::info
This probe can only be triggered once per minute.
:::

#### Test 2: Block a real attack

Try to access a `.env` file — a common technique attackers use to steal credentials left on servers. This is covered by the `crowdsecurity/appsec-virtual-patching` collection.

```bash
curl -i http://<your-service-url>/.env
```

Expected result — the WAF returns **HTTP 403 Forbidden**:

```
HTTP/1.1 403 Forbidden
...
```

#### Test 3: SQL injection (requires `crowdsecurity/appsec-generic-rules`)

Simulate a SQL injection attack with a scanner-like user agent:

```bash
curl -i -H "User-Agent: Nikto/2.1.6" "http://<your-service-url>/?id=1+UNION+SELECT+1,2,3--"
```

Expected result: **HTTP 403 Forbidden** — the request is blocked inline.

#### View blocked attacks

After triggering attacks, check what the WAF caught:

```bash
# List recent AppSec alerts
sudo cscli alerts list

# Show per-rule block counts
sudo cscli metrics show appsec
```

For a richer view with attack timelines and rule breakdowns, open the **[CrowdSec Console → Alerts](https://app.crowdsec.net)** tab — AppSec blocks appear automatically with full context.

![appsec-console](/img/appsec_console.png)

## Optional: Advanced Configuration

### Multiple AppSec Configurations

You can [load multiple AppSec configurations](/appsec/configuration_creation_testing.md#defining-multiple-appsec-configurations) for different rule sets:

```yaml
# /etc/crowdsec/acquis.d/appsec.yaml
appsec_configs:
  - crowdsecurity/appsec-default    # Virtual patching rules (in-band)
  - crowdsecurity/crs               # OWASP CRS rules (out-of-band)
labels:
  type: appsec
listen_addr: 127.0.0.1:7422
source: appsec
name: myAppSecComponent
```

### Custom Port Configuration

To use a different port, update the `listen_addr` in your acquisition file and ensure your remediation component points to the same address.

## Troubleshooting

If the AppSec Component fails to start:

1. **Check port availability**: Ensure port 7422 isn't already in use
2. **Verify collections**: Run `sudo cscli collections list` to confirm installation
3. **Check configuration syntax**: Validate your `appsec.yaml` file
4. **Review logs**: Check `/var/log/crowdsec.log` for error messages

For detailed troubleshooting, see the [AppSec Troubleshooting Guide](/appsec/troubleshooting.md).
